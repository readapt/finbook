## 1. 需求完整性评审

### 1.1 MVP 范围评估

**优点**:
- MVP 分 4 个版本迭代 (v0.1-v0.4)，范围清晰
- 核心功能聚焦于凭证管理、银行对账、资产管理、财务报表
- 优先级标注合理 (P0/P1)

**遗漏/不足**:

| 类别 | 遗漏项 | 重要性 |
|------|--------|--------|
| 基础账务 | 手工凭证录入、凭证审核流程 | 🔴 高 |
| 税务 | 增值税计算逻辑、进项销项管理 | 🔴 高 |
| 多币种 | 外币业务处理、汇率管理 | 🟡 中 |
| 权限 | 多用户角色 (会计/出纳/管理员) | 🔴 高 |
| 对账 | 往来对账 (应收/应付) | 🟡 中 |
| 备份 | 自动化备份恢复机制 | 🔴 高 |

**关键遗漏**: 财政部规范要求会计软件必须有**凭证审核流程**，当前设计缺失。

### 1.2 用户角色缺失

当前文档假设单用户场景，但实际企业至少需要：
- **制单人** → **审核人** → **记账人** 分离（财政部规范要求）

## 2. 技术可行性评审

### 2.1 Beancount + 自研业务层架构

**架构合理性**: ✅ **基本合理**

```
┌─────────────────┐
│   业务逻辑层     │ ← 自研
├─────────────────┤
│ 中国化合规层     │ ← 自研（关键）
├─────────────────┤
│  Beancount      │ ← 开源
└─────────────────┘
```

**优点**:
- 利用 Beancount 成熟的双式记账核心
- 文本存储便于审计和版本控制
- 分层设计清晰

**技术挑战**:

| 挑战 | 难度 | 说明 |
|------|------|------|
| Beancount v3 稳定性 | 🟡 中 | v3 仍在开发中，API 可能变化 |
| 中国会计科目映射 | 🔴 高 | 需建立完整科目体系 |
| 并发写控制 | 🟡 中 | 文件锁在高并发下有瓶颈 |
| 双存储一致性 | 🟡 中 | 最终一致性可能短暂数据不同步 |

### 2.2 OpenClaw 依赖风险

**风险点**:
- OpenClaw 是较新的框架，社区成熟度待验证
- 企业微信/钉钉集成文档未提及具体实现方案
- Telegram 在国内使用受限（验证期方案不可用于生产）

## 3. 合规性评审

### 3.1 财政部 2025 规范对照

| 规范要求 | 文档实现 | 状态 |
|----------|----------|------|
| 凭证不可逆 | 凭证只追加，不修改 | ✅ 已考虑 |
| 凭证连续编号 | 提及"连续编号" | ⚠️ 需实现 |
| 电子会计凭证 | 支持 PDF/OFD/XML | ⚠️ OFD 未明确 |
| 操作留痕 | 审计日志 | ✅ 已考虑 |
| 数据导出 | CSV/Excel | ✅ 已考虑 |
| **审核流程** | **未提及** | ❌ **缺失** |
| **权限分离** | **未提及** | ❌ **缺失** |
| 境内数据存储 | 未明确 | ⚠️ 需说明 |

**合规风险**: 当前设计**不满足**财政部规范的完整要求，需补充审核流程和权限管理。

### 3.2 金税四期对接

| 能力 | 状态 |
|------|------|
| 发票验真 API | ⚠️ 仅提及，未实现 |
| 数电票支持 | ⚠️ 仅提及 PDF+XML |
| 三流一致校验 | ❌ 未提及实现方案 |

## 4. 风险识别

### 4.1 技术风险

| 风险 | 等级 | 影响 | 建议 |
|------|------|------|------|
| Beancount v3 API 不稳定 | 🟡 中 | 需频繁适配 | 考虑锁定版本或抽象接口 |
| OCR 准确率不足 | 🟡 中 | 用户体验差 | 增加人工校验环节 |
| 文件锁并发瓶颈 | 🟡 中 | 多用户场景性能下降 | 评估 PostgreSQL 作为主存储 |
| 文本文件损坏 | 🟡 中 | 数据丢失 | 增加 WAL 日志机制 |
| OpenClaw 依赖过深 | 🟡 中 | 切换成本高 | 抽象入口层接口 |

### 4.2 业务风险

| 风险 | 等级 | 影响 | 建议 |
|------|------|------|------|
| 财务数据敏感，用户不信任自托管 | 🟡 中 | 获客困难 | 提供托管服务选项 |
| Telegram 国内不可用 | 🔴 高 | 验证期方案不可行 | 优先开发企业微信 |
| 代账公司被排除，市场缩小 | 🟡 中 | 收入受限 | 重新评估定价策略 |
| 竞品已有成熟方案（如云帐房） | 🟡 中 | 竞争激烈 | 差异化定位需明确 |

### 4.3 法律风险

| 风险 | 等级 | 说明 |
|------|------|------|
| AGPL 传染性 | 🟡 中 | 修改 Beancount 需开源 |
| 数据跨境风险 | 🟡 中 | 如服务器在境外违反《数据安全法》 |
| 财务软件资质 | 🟡 中 | 是否需软件企业认证 |

## 5. 改进建议

### 5.1 架构建议

| 建议 | 优先级 | 说明 |
|------|--------|------|
| 抽象存储层接口 | 🔴 高 | 支持 SQLite/PostgreSQL/文本灵活切换 |
| 增加消息队列 | 🟡 中 | 异步处理 OCR、报表生成 |
| 增加 API 层 | 🟡 中 | 便于未来 Web 端集成 |
| 引入配置中心 | 🟡 中 | 管理科目表、折旧政策等 |

**建议调整架构图**:

```
入口层 → API Gateway → 业务服务层 → 合规层 → 存储层
                              ↓
                      异步任务队列 (OCR/报表/备份)
```

### 5.2 功能建议

**必须补充**:
1. **凭证审核流程** - 制单→审核→记账
2. **角色权限系统** - 管理员/会计/出纳/只读
3. **红字冲销** - 实现不可逆记账
4. **科目体系** - 内置小企业会计准则科目表
5. **自动备份** - 定时备份到本地/云存储

**建议补充**:
- 多账套管理（适合代账场景）
- 凭证模板（常用业务快速录入）
- 移动端 H5（替代 Telegram）

### 5.3 实施建议

| 阶段 | 建议 |
|------|------|
| **短期** | 1.补充合规层设计 2.优先开发企业微信入口 3.实现基本审核流程 |
| **中期** | 1.对接税局发票 API 2.完善报表体系 3.建立测试案例库 |
| **长期** | 1.申请软件产品认证 2.建立合作伙伴生态 3.通过安全审计 |

## 结论

**项目整体可行，但存在以下关键问题需优先解决**:

1. 🔴 **合规缺口** - 必须补充凭证审核、权限管理
2. 🔴 **入口风险** - Telegram 不可用于国内生产环境
3. 🟡 **架构优化** - 评估文件锁在高并发场景的可行性
4. 🟡 **差异化定位** - 与现有财务软件竞品差异化不够清晰

**建议**: 先完成合规层详细设计，再启动开发。优先保证合规性，再追求功能丰富度。
